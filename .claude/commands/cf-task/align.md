# cf-task:align

从需求出发，通过结构化对话产出设计简报（`.design.md`），为 `cf-task:plan` 提供足够丰富的输入。支持三种输入：纯文本需求、`.prd.md` 产品需求文档、`.design.md` 草稿。

## 输入

- `/cf-task:align "需求描述"` — 从需求描述新建设计简报
- `/cf-task:align <.prd.md 路径>` — 基于已有 PRD 派生设计简报（推荐路径）
- `/cf-task:align <.design.md 路径>` — 恢复已有草稿继续讨论
- `/cf-task:align` — 交互式，先问用户要做什么

## 执行步骤

### 1. 确定模式

按参数类型识别（顺序判断，优先匹配更具体的后缀）：

| 参数形态 | 模式 | 行为 |
|---------|------|------|
| 已存在的 `.design.md` 文件 | **恢复模式** | Read 读取，展示当前内容，询问调整点，进入 Step 3 |
| 已存在的 `.prd.md` 文件 | **PRD 派生模式** | Read PRD，执行 Step 2.6 预填充维度，Step 3 只补技术维度 |
| 已存在的其他 `.md` 文件 | **新建模式（带上下文）** | Read 作为参考材料，进入 Step 2 |
| 纯文本字符串 | **新建模式** | 以文本作为初始需求，进入 Step 2 |
| 无参数 | **交互模式** | 询问"要实现什么功能？"，进入 Step 2 |

### 2. 代码库上下文扫描

快速了解项目现状，为后续建议提供依据：

1. 用 Glob 检查 `.code-flow/specs/*/_map.md` 是否存在，若有则 Read 读取（了解项目结构）
2. 检测技术栈：Read `package.json` / `requirements.txt` / `go.mod` / `pyproject.toml` 等（取存在的）
3. 用 Glob 扫描与需求相关的现有代码目录，了解已有模式

目的：**让后续建议基于实际代码库**，而非泛泛的通用建议。

### 2.5. 识别需求类型，选择维度

根据需求描述和代码库上下文，判断本次工作的性质，选择适用的讨论维度。

**模板选择规则**：
- **design-lite.md** — 适用于：功能开发(简单) | CLI/脚本 | Bug修复 | 小型重构
- **design-full.md** — 适用于：跨系统集成 | 性能优化 | 架构演进 | 中大型功能开发

AI 应根据需求复杂度主动建议使用哪个模板。默认优先 lite 版本，用户要求完整设计时使用 full 版本。

**模板路径**：
- Lite: `.code-flow/specs/shared/design/design-lite.md`
- Full: `.code-flow/specs/shared/design/design-full.md`

先读取对应模板文件作为生成基准。

**核心维度（所有类型必选）**：
- 目标与边界
- 技术方案
- 约束条件
- 验收标准

**扩展维度（按需选择，不适用则跳过）**：
- 数据模型 — 涉及数据库/存储变更时
- 接口设计 — 涉及 API / CLI / 函数入口变更时

常见需求类型与维度匹配：

| 需求类型 | 数据模型 | 接口设计 | 技术方案侧重 |
|----------|---------|---------|-------------|
| 功能开发（含 API） | ✓ | ✓ API | 组件设计、第三方库选型 |
| 功能开发（纯后台） | 视情况 | 视情况 | 模块结构、算法/流程、数据流 |
| CLI 命令/脚本 | — | ✓ CLI | 命令结构、参数、输出格式 |
| 重构 | — | — | 变更范围、迁移路径、兼容性 |
| 性能优化 | — | — | 瓶颈分析、优化策略、度量指标 |
| 定时任务/队列 | 视情况 | — | 调度策略、失败处理、幂等性 |
| 日志/监控 | — | — | 采集点、指标定义、告警规则 |
| 构建/CI | — | — | 流水线设计、触发条件、产物 |
| 缺陷修复 | 视情况 | 视情况 | 根因分析、修复方案、回归范围 |

> 上表仅供参考，AI 应根据实际需求灵活判断，不机械套用。

### 2.6. PRD 派生处理（仅 PRD 派生模式）

从 PRD 各章节提取结构化信息，映射到设计维度作为 Step 3 的预填充上下文：

| PRD 章节 | 映射到设计维度 | 处理方式 |
|---------|--------------|---------|
| §2 背景与目标 | 目标与边界（问题/目标） | 直接继承 |
| §3 用户与场景（含 US-XX） | 目标与边界（用户） | 继承用户画像；US 列表用于 FEAT 追溯 |
| §4 功能需求（FEAT-XX） | 功能方案（FEAT 清单） | 继承 FEAT 清单与优先级；保留 US→FEAT 来源映射 |
| §5 非功能需求 | 约束条件（性能/安全/兼容） | 直接继承 |
| §6 范围与边界 | 目标与边界（In/Out Scope） | 直接继承 |
| §7 依赖与风险 | 风险与依赖 | 直接继承 |

**需向用户呈现的摘要**：
```
已从 PRD 解析：
  - 功能 N 个（P0: X, P1: Y）
  - 用户故事 M 个
  - 非功能约束 K 条
  - 范围内/外清单已继承

本轮将聚焦技术维度：数据模型 / 接口设计 / 技术方案
PRD 已覆盖的维度如无需调整，将直接使用。
```

**关键约束**：
- Step 3 中已从 PRD 获得充分信息的维度**不重复提问**，除非用户主动修改
- design 中 FEAT 的"来源"列必须引用 PRD 中对应的 US-XX

### 3. 交互式细化

围绕选中的维度逐步推进，**每轮提 2-3 个问题**（不要一次性抛 10 个问题）：

**目标与边界**（必选；PRD 派生模式下已覆盖，只需用户确认）
- 解决什么问题？为谁？
- 做什么？（核心功能列表）
- 不做什么？（明确排除项）

**数据模型**（涉及数据库/存储变更时）
- 需要哪些表/集合？
- 字段、类型、约束、索引？
- 与现有模型的关系？（外键、关联）

**接口设计**（涉及 API / CLI / 函数入口时）

API 场景：
- 哪些 endpoint？请求/响应格式？错误码？

CLI 场景：
- 命令名称和子命令？参数和 flag？输出格式？

内部函数入口场景：
- 公开接口签名？输入输出类型？错误处理约定？

**技术方案**（必选，内容随需求类型变化）
- 功能开发：关键组件、模块结构、第三方库选型、与现有架构集成方式
- 重构：变更范围、分步迁移路径、向后兼容策略
- 性能优化：瓶颈定位、优化策略、性能度量方式
- 定时任务：调度策略、失败重试、幂等性保证
- 日志/监控：采集点、指标定义、告警阈值
- 构建/CI：流水线阶段、触发条件、产物和部署
- 缺陷修复：根因分析、修复方案、回归测试范围

**约束条件**（必选；PRD 派生模式下已从 NFR 继承）
- 技术栈、性能要求、兼容性、时间线等

**验收标准**（必选）
- 怎么判断做完了？
- 可验证的条件列表

**交互原则**：
- 基于 Step 2 扫描到的代码库上下文**提出具体默认建议**，而非空泛提问
- 每轮回答后更新理解，追问细节或推进到下一个维度
- 涉及数据库和接口的维度**必须具体到表结构和请求/响应 schema**
- 不涉及数据库和接口时，**不要强制展开这些维度**
- PRD 派生模式下，已从 PRD 覆盖的维度不再提问；用户主动要求调整除外
- 用户说"你定"或"随意"时，根据代码库上下文选择最合理的默认方案，明确告知用户采用了什么
- 输出语言默认与用户需求语言保持一致；若用户未明确指定语言，默认使用中文生成完整设计简报（标题、正文、表头均为中文）
- 仅当用户明确要求英文或项目已有强约定要求英文文档时，才输出英文；避免中英混杂

**中断恢复**：在进入 Step 3 之前**提前确定文件路径**（Step 5 的命名与目录逻辑）。当任意两个维度已有实质内容时，立即写入草稿文件。后续每轮对话更新同一文件。这样即使会话中断，进度不会丢失。

### 4. 基于模板生成设计简报

选中的维度都有实质内容后，基于 Step 2.5 选择的模板文件生成设计简报。

**操作步骤**：
1. 用 Read 读取选定的模板文件（design-lite.md 或 design-full.md）
2. 按照模板章节结构，将 Step 3 中收集的内容填入对应位置
3. **只包含适用的章节**，模板中有但需求不涉及的章节跳过
4. 用 `{}` 占位符替换为实际内容
5. PRD 派生模式：功能清单的"来源"列填 PRD 中对应的 US-XX
6. 展示生成的设计简报

**章节映射**（Lite 模板，对应 `design-lite.md` 实际章节）：

| 模板章节 | 内容来源 |
|---------|---------|
| 1. 文档控制 | 自动生成（修订历史 v0.1） |
| 2.1 需求概述 | Step 3 的"目标与边界"（PRD 派生时继承自 PRD §2） |
| 2.2 功能方案 | Step 3 的功能清单 + 字段约束（FEAT 继承自 PRD §4，"来源"列填 US-XX） |
| 2.3 范围与边界 | Step 3 的 In/Out Scope（PRD 派生时继承自 PRD §6） |
| 2.4 验收条件 | Step 3 的"验收标准"（正常/异常场景 + 非功能指标） |
| 3.1 技术选型 | Step 3 的"技术方案"（语言/框架/数据库选型） |
| 3.2 架构设计 | Step 3 的"技术方案"（分层与组件，按需） |
| 3.3 接口设计 | Step 3 的"接口设计"（API/CLI/函数签名，按需） |
| 4. 风险与依赖 | Step 3 的依赖与风险（PRD 派生时继承自 PRD §7） |

**章节映射**（Full 模板，对应 `design-full.md` 实际章节）：

| 模板章节 | 内容来源 |
|---------|---------|
| 1. 文档控制 | 自动生成 |
| 2.1 需求概述 | PRD §2 模块/背景/核心目标 |
| 2.2 痛点与价值 | PRD §2 + §3（目标用户、当前问题、业务影响、预期价值、US 清单） |
| 2.3 功能方案 | Step 3 的功能清单 + 字段约束（FEAT 继承自 PRD §4，"来源"列填 US-XX） |
| 2.4 范围与边界 | Step 3 的范围（继承自 PRD §6） |
| 2.5 验收条件 | Step 3 的"验收标准"（业务规则 RULE + 场景 S/E/B + NFR） |
| 3.1 方案选型 | Step 3 的"技术方案"（备选对比 + 技术栈） |
| 3.2 架构设计 | Step 3 的"技术方案"（架构图 + 分层 + 外部依赖） |
| 3.3 数据设计 | Step 3 的"数据模型"（表 + 索引 + ER + 容量） |
| 3.4 接口设计 | Step 3 的"接口设计" |
| 3.5 质量实现方案 | Step 3 的"约束条件"（性能/可靠性/安全/可观测性落地） |
| 4. 部署与运维 | 按需（发布回滚、监控告警、数据迁移） |
| 5. 风险与依赖 | Step 3 的依赖与风险（继承自 PRD §7） |
| 6. 需求追溯矩阵 | 自动生成（US→FEAT→API→TC 关联） |

展示后询问用户：**确认写入？可以调整任何部分。**

### 5. 写入文件

用户确认后：

1. **确定文件名**：
   - **PRD 派生模式**：继承 PRD 文件名（去 `.prd` 后缀）。例：`user-login-flow.prd.md` → `user-login-flow`
   - **恢复模式**：保持原文件名
   - **新建/交互模式**：从目标推导 kebab-case，如 `user-auth`
2. **确定目录**：
   - **PRD 派生模式**：使用 PRD 所在目录，保证 PRD 与 design 同目录（便于 `cf-task:archive` 配对归档）
   - **恢复模式**：保持原目录
   - **新建/交互模式**：按当前日期创建 `.code-flow/tasks/<YYYY-MM-DD>/`
3. 根据选择的模板确定文件后缀：
   - Lite 模板: `<name>-lite.design.md`
   - Full 模板: `<name>.design.md`
4. 用 Write 写入最终路径

### 6. 输出摘要

```
已生成设计简报: <完整路径>
模板类型: Lite / Full
来源: <PRD 派生 / 新建 / 恢复>

内容:
  - <按实际包含的章节列出>
  - 验收标准: N 项

下一步:
  - 审阅/编辑: 直接打开设计简报文件
  - 标注问题: 在文件中添加 #NOTES，然后 /cf-task:note <name>.design
  - 生成任务: /cf-task:plan <完整路径>
```
